# 260619 - типы потоков разработки - фича-бриф

## Заказчик

Техлид фреймворка.

## Контекст

Сейчас фреймворк задаёт единый поток разработки фичи.
Нужно переосмыслить его как набор типов работ.
Эта итерация сама ведётся как универсальная задача: сначала фиксируем целевое состояние, затем отдельно изучаем текущее устройство, затем отдельно выбираем направление перехода.

## Целевые изменения

1. Универсальный поток `Задача` задаёт базовую последовательность работы: понять целевое состояние, понять текущее состояние, выбрать общее направление перехода, при необходимости подготовиться, выполнить работу.
2. Потоки для конкретных типов работ рассматриваются как специализации универсального потока, если это подтвердится в дальнейшем анализе.
3. Поток реализации feature остаётся одним из специализированных потоков, но не выделяется как особый user-facing-only базовый сценарий.
4. Рефакторинг выделяется в отдельный тип потока.
5. Рефакторинг может входить в feature-поток как фаза `Предварительный рефакторинг`.
6. Для bug fix репродьюсер пишется во второй фазе: при понимании текущего устройства и локализации дефекта.
7. Для bug fix предварительный рефакторинг может потребоваться уже во второй фазе, если без него невозможно или неэргономично написать репродьюсер.
8. Пример универсальной задачи: провести нагрузочное тестирование.

## Точки наблюдения

- Framework workflow: правила и артефакты, которые задают последовательность работы над задачами, фичами и рефакторингом.

## Критерии приёмки

1. Фреймворк явно различает универсальный поток задачи и специализированные потоки работ.
2. Универсальный поток описывает обязательные фазы: целевое состояние, текущее состояние, направление перехода, выполнение.
3. Универсальный поток допускает опциональную фазу подготовки между направлением перехода и выполнением.
4. Рефакторинг можно вести как самостоятельную работу и как предварительную фазу feature-потока.
5. Bug fix guidance относит написание репродьюсера к фазе понимания текущего устройства.
6. Bug fix guidance допускает предварительный рефакторинг внутри фазы понимания текущего устройства, когда это нужно для написания репродьюсера.
7. Feature-поток не закрепляется как исключительно user-facing сценарий без отдельного решения.
